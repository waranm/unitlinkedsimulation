Daily → Monthly Resampling

Simulation วิ่งทีละ 1 เดือน ดังนั้นต้องแปลงข้อมูล daily เป็น monthly ก่อน โดยเก็บ ค่า NAV สุดท้ายของแต่ละเดือน (end-of-month) ถ้าไม่ทำขั้นตอนนี้ mean return จะถูก understate ประมาณ 21 เท่า 

// ตรวจ: ถ้า avg gap < 20 วัน = daily data → ต้อง resample
const avgGapDays = spanDays / (rows.length - 1);
if (avgGapDays >= 20) return rows; // monthly แล้ว

// Group by YYYY-MM → เก็บ row สุดท้ายของแต่ละเดือน
const byMonth = new Map();
for (const row of rows) {
  const key = `\${year}-\${month}`;
  byMonth.set(key, row); // later row overwrites → end-of-month
}

Log-Return

ใช้ log-return แทน arithmetic return เพราะ log-return มีคุณสมบัติ additive ตามเวลา และ assume distribution เป็น Normal ได้ดีกว่า

r_t = ln(NAV_t / NAV_{t−1})

μ̂ = (1/T) · Σ r_t     ← mean log-return รายเดือน
σ̂² = (1/(T−1)) · Σ (r_t − μ̂)²   ← sample variance (Bessel's correction)
σ̂ = √σ̂²     ← monthly std
Bessel's Correction (หาร n−1 ไม่ใช่ n)

ข้อมูลย้อนหลังคือ ตัวอย่าง ไม่ใช่ประชากรทั้งหมด การหาร n ให้ variance ที่ต่ำกว่าจริง (biased downward) การหาร n−1 แก้ค่าเบี่ยงเบนนี้ และเป็น unbiased estimator ของ population variance

Offer/BID Ratios

เฉลี่ยอัตราส่วน offer/nav และ bid/nav จากข้อมูลทุก row แล้วนำ ratios เหล่านี้ไปปรับ simulated NAV ทุกเดือนขณะ simulation

offerRatio = avg(offer_t / nav_t) ← ใช้คูณ NAV เมื่อซื้อหน่วย
bidRatio = avg(bid_t / nav_t) ← ใช้คูณ NAV เมื่อประเมินมูลค่า

ทำไมต้องใช้ Correlation

ถ้า simulate แต่ละกองทุนแบบ independent ความผันผวนของ portfolio จะต่ำกว่าความเป็นจริง เพราะกองทุน equity มักเคลื่อนไหวไปในทิศทางเดียวกันในช่วงที่ตลาดผันผวน การ ignore correlation ทำให้ diversification ถูก overestimate

ขั้นตอน — Correlation Rebuild Strategy
Step 1 — ประมาณ σ_i จาก history ของกองทุน i ทั้งหมด
  ใช้ full history ของแต่ละกอง (ไม่ตัดให้เท่ากัน)
  ← ได้ variance estimate ที่ unbiased ต่อแต่ละกอง

Step 2 — ประมาณ ρ_ij จาก overlap จริงของคู่ (i, j) เท่านั้น
  ใช้ pairwise overlapping returns (weekly → monthly fallback)
  พร้อม shrinkage และ MIN_OBS guard
  ← รายละเอียดใน Section ถัดไป

Step 3 — Rebuild Σ จาก σ และ ρ
  Σ_ij = ρ_ij · σ_i · σ_j   ← internally consistent

Step 4 — Cholesky Decomposition
  หา lower-triangular L ที่ทำให้ L · Lᵀ = Σ

Step 5 — Correlated Shocks ใน Simulation
  z ~ N(0, I) → w = L · z → Cov(w) = L·Lᵀ = Σ
ใน Simulation
// ทุกเดือน: draw independent standard normals
const z = Array.from({ length: nFunds }, randNormal); // z ~ N(0,I)

// Multiply by Cholesky L → correlated shocks
for (let i = 0; i < nFunds; i++) {
  let shock = 0;
  for (let k = 0; k <= i; k++) shock += L[i][k] * z[k]; // w = L·z
  nav[funds[i]] *= Math.exp(μ_i + shock); // GBM update
}
✓
กรณี Single Fund: L = [[σ]] — เหมือนกับ GBM ปกติทุกประการ ไม่มี overhead จาก Cholesky

ปัญหาของ Approach เดิม

approach เดิมใช้ inception backfill — กองทุนที่เพิ่งเปิดตัวจะถูก ยัดค่า NAV แรกย้อนไปจนถึงวันเริ่มต้นของกองที่เก่าที่สุด ทำให้ covariance matrix มี 2 ปัญหาใหญ่:

⚠️
Variance ถูก Deflate
ช่วงที่ backfill ให้ log-return = 0 ทุกเดือน (ln(NAV/NAV) = 0) ทำให้ T พองตัว แต่ variance กลับต่ำกว่าจริง
⚠️
Correlation ถูก Attenuate
ช่วง backfill ที่ return = 0 ทำให้ correlation ระหว่างกองถูก ดึงเข้าหา 0 โดยไม่มี statistical basis
กลยุทธ์ใหม่ — Correlation Rebuild

แยก 2 parameter ออกจากกัน แล้ว rebuild covariance จาก component:

σ_i ← ประมาณจาก full history ของกอง i (ไม่แตะ)
ρ_ij ← ประมาณจาก overlapping periods จริงของคู่ (i,j) เท่านั้น

Σ_ij = ρ_ij · σ_i · σ_j   ← internally consistent, unbiased

วิธีนี้ทำให้กองที่เปิดมานาน 10 ปีได้ variance ที่ถูกต้องจากข้อมูลทั้งหมด ขณะที่กองใหม่ยังได้ correlation ที่ประมาณจากช่วงที่มีข้อมูลจริง

Weekly → Monthly Grain Selection

ทุก pair ลองใช้ weekly returns ก่อน (ข้อมูลหนาแน่นกว่า ลด Epps bias) ถ้า overlap < 26 สัปดาห์ (~6 เดือน) ตกลงไปใช้ monthly returns

ลำดับ:
1. สร้าง YYYY-Www → NAV map ต่อกอง (ISO-week key, ใช้ Thursday เป็น anchor)
2. หา intersection ของ weeks ที่มีข้อมูลทั้ง 2 กอง
3. คำนวณ log-return เฉพาะ consecutive weeks (ข้าม gap ถ้าขาด)
4. ถ้า T_weekly < 26 → ทำซ้ำขั้นตอน 1-3 ด้วย monthly maps
5. ถ้า T_monthly < 6 → ρ_ij = 0 (MIN_OBS guard — ดูส่วนถัดไป) 
// cov-builder.js — grain selection per pair
let { ra, rb, T } = pairwiseReturns(weeklyMaps[i], weeklyMaps[j]);
if (T < MIN_OBS_WEEKLY) {          // MIN_OBS_WEEKLY = 26
  ({ ra, rb, T } = pairwiseReturns(monthlyMaps[i], monthlyMaps[j]));
  pairGrain = 'monthly';
}
MIN_OBS Guard — ป้องกัน Noisy Correlation

ถ้า overlap สั้นเกินไป correlation ที่ได้จะ noisy มาก (เช่น 3 เดือน ρ จาก 3 data points มี error ±0.5 ขึ้นไป) ดีกว่าตั้ง ρ_ij = 0 ตรงๆ แทนที่จะใส่ค่า noisy เข้า matrix

MIN_OBS_WEEKLY = 26 สัปดาห์ (~6 เดือน)
MIN_OBS_MONTHLY = 6 เดือน

ถ้า overlap < threshold ทั้ง weekly และ monthly → ρ_ij = 0
(จำนวน zeroed pairs รายงานใน diagnostics.zeroedPairs)
Per-Pair Shrinkage toward Identity

แม้ overlap ผ่าน MIN_OBS แต่ถ้าสั้น correlation estimate ยังมี noise สูง ใช้ Ledoit-Wolf-style shrinkage ดึง ρ เข้าหา 0 ตาม ความสั้นของ overlap (per-pair — ไม่ใช่ per-matrix)

δ_ij = min(1, (N+1) / (T_monthly − 1))

ρ_shrunk = (1 − δ_ij) · ρ_sample

N = จำนวนกองทุนทั้งหมด
T_monthly = overlap เป็นเดือน (weekly ÷ 4.33)

δ → 0 เมื่อ T ยาว   (overlap มาก → เชื่อ sample correlation)
δ → 1 เมื่อ T สั้น หรือ N ใหญ่  (ดึงเข้าหา 0 เต็มๆ)

ตัวอย่าง: N=5 กองทุน, overlap 12 เดือน → δ = 6/11 ≈ 0.55 → ρ_shrunk ≈ 0.45 · ρ_sample (ดึงเข้าหา 0 ครึ่งหนึ่ง)

PSD Repair — Eigenvalue Clipping

การ shrink แต่ละ pair ด้วย δ ที่ต่างกัน อาจทำให้ correlation matrix ไม่ Positive Semi-Definite (PSD) ซึ่ง Cholesky decomposition จะ fail วิธีแก้: Jacobi eigendecomposition + eigenvalue clipping

1. Eigen-decompose: C = V · Λ · Vᵀ   (Jacobi rotation, stable สำหรับ N ≤ 200)
2. Clip eigenvalues: λ_i = max(λ_i, 1e-8)
3. Reconstruct: M = V · Λ_clipped · Vᵀ
4. Rescale diagonal → 1   (คืนรูป correlation matrix)
5. Enforce symmetry: M_ij = M_ji = avg(M_ij, M_ji)
✓
ถ้า matrix PSD อยู่แล้วจะ return ของเดิมเลย ไม่มี computational overhead (psdRepaired = false ใน diagnostics)

สรุป Decision Tree
สำหรับแต่ละ pair (i, j):

  1. หา weekly overlap T_w
  2. ถ้า T_w ≥ 26 → ใช้ weekly, Tmonthly = T_w / 4.33
  3. ถ้า T_w < 26 → ลอง monthly T_m
      ถ้า T_m ≥ 6 → ใช้ monthly, Tmonthly = T_m
      ถ้า T_m < 6 → ρ_ij = 0 (zeroed pair)
  4. คำนวณ Pearson ρ จาก returns ที่ได้
  5. Shrink: ρ_shrunk = (1 − δ) · ρ โดย δ = min(1, (N+1)/(Tmonthly−1))
  6. ใส่ ρ_shrunk ลงใน correlation matrix

หลัง loop ทุก pair:
  7. PSD repair (ถ้า matrix ไม่ PSD)
  8. Rebuild: Σ_ij = ρ_ij · σ_i · σ_j
UL Product Constraints และผลต่อ Shrinkage

ผลิตภัณฑ์ Unit Linked นี้มี กฎ allocation ที่กำหนดโดย product rule:

📦 จำนวนกองทุน
เลือกได้ 1–10 กอง ต่อกรมธรรม์
ไม่เกิน 10 กอง
📊 สัดส่วนการลงทุน
แต่ละกองต้องได้รับ ≥ 5% ของเบี้ย
เพื่อป้องกัน allocation ที่ไม่มีนัยสำคัญ

กฎทั้งสองนี้มีผลโดยตรงต่อ ความน่าเชื่อถือของ covariance matrix ที่ใช้ใน simulation ผ่านสมการ shrinkage:

δ_ij = min(1, (N+1) / (T_monthly − 1))

N = จำนวนกองทุนที่ user เลือก   ← ถูก bound ด้วย product rule
T_monthly = overlap เป็นเดือน

ถ้า N ใหญ่เกินไป → δ → 1 → correlation ทุก pair ถูก shrink เข้าหา 0 ทั้งหมด
ถ้า N ≤ 10 → δ อยู่ในช่วงที่สมเหตุสมผล

ตารางต่อไปนี้แสดงค่า δ สำหรับทุก combination ที่เป็นไปได้ภายใต้ product rule โดยใช้ T ≈ 40 เดือน (กองทุนเปิดมาตั้งแต่ Dec 2022 ถึงปัจจุบัน):

Math
Geometric Brownian Motion (Discrete Time)

NAV แต่ละเดือนถูก simulate ตามโมเดล GBM แบบ discrete:

NAV_t = NAV_{t−1} · exp(μ + w_t)

เมื่อ w_t ~ N(0, σ²) สำหรับกองทุนเดียว
หรือ w = L · z สำหรับหลายกองทุน (correlated)

μ = monthly log-return mean (ประมาณจากข้อมูลจริง)
σ = monthly log-return std (ประมาณจากข้อมูลจริง)
Box-Muller Transform — การสุ่มค่า Normal

ใช้ Box-Muller transform แปลง Uniform random → Standard Normal ซึ่งถูกต้องทางสถิติและเร็วกว่า Central Limit Theorem approximation

U₁, U₂ ~ Uniform(0,1)
Z = √(−2 ln U₁) · cos(2π U₂) → Z ~ N(0,1)
Volatility Drag (Itô Correction)

เนื่องจากโมเดลใช้ log-return mean (μ) โดยตรง arithmetic drift จริงของ NAV คือ exp(μ + σ²/2) ดังนั้น μ = 0 หมายถึง arithmetic drift เป็นบวก ไม่ใช่ zero ในทางกลับกัน median NAV จะลดลงตามเวลาเมื่อ μ = −σ²/2 (zero arithmetic drift)

E[NAV_t / NAV_0] = exp(μ + σ²/2) · t   ← arithmetic mean path
median(NAV_t / NAV_0) = exp(μ · t)   ← median path (lower)

ผล: distribution ของ final value เป็น right-skewed
→ ค่าเฉลี่ย (mean) > ผลลัพธ์ทั่วไป (median) เสมอเมื่อ σ > 0

ทำไมต้องใช้ Regime Switching

โมเดล GBM พื้นฐานใช้ μ และ σ คงที่ตลอด simulation แต่ตลาดจริงสลับระหว่าง "สภาวะ" ที่แตกต่างกัน เช่น Bull market, Bear market, และ Crisis แต่ละสภาวะมี return และความผันผวนต่างกัน และแนวโน้มที่จะอยู่ในหรือออกจากสภาวะนั้นก็ต่างกัน Regime Switching model จำลองพฤติกรรมนี้ด้วย Markov Chain ทำให้ downside scenarios สมจริงยิ่งขึ้น (fat tail, volatility clustering)

Markov Chain — หลักการ

ทุกเดือน simulation อยู่ใน regime หนึ่งเสมอ ความน่าจะเป็นในการเปลี่ยน regime ขึ้นกับเฉพาะสภาวะปัจจุบันเท่านั้น (Markov property) ไม่ใช่ประวัติการณ์ทั้งหมด

P(regime ถัดไป = j | regime ปัจจุบัน = i) = P[i][j]

ตัวอย่าง transition matrix (3 regimes):

            → Bull  Bear  Crisis
Bull   [ 0.90  0.09  0.01 ]  ← อยู่ Bull มีโอกาส 90% ยังอยู่ Bull ต่อ
Bear   [ 0.15  0.75  0.10 ]  ← Bear มีโอกาส 15% ฟื้นเป็น Bull
Crisis [ 0.25  0.45  0.30 ]  ← Crisis มักเปลี่ยนเป็น Bear ก่อน
Regime Parameters — muScale และ sigmaScale

แต่ละ regime ปรับ scale ค่า μ และ σ ที่ประมาณจากข้อมูลจริง แทนการกำหนดค่าสัมบูรณ์ วิธีนี้ทำให้ใช้ได้กับทุกกองทุนโดยไม่ต้องรู้ค่าจริงล่วงหน้า

NAV_t = NAV_{t−1} · exp(μ̂ · muScale + w_t · sigmaScale)

muScale > 1   → return สูงกว่าค่าเฉลี่ยประวัติศาสตร์ (Bull)
muScale < 0   → market ติดลบ (Bear / Crisis)
sigmaScale > 1 → ความผันผวนสูงกว่าปกติ
sigmaScale < 1 → ตลาดนิ่ง, ผันผวนน้อย
Stationary Distribution — การตั้งต้น Regime

ต้นแต่ละ scenario จะสุ่ม regime เริ่มต้นจาก stationary distribution π ซึ่งคือสัดส่วนเวลาที่ chain ใช้ใน regime นั้นในระยะยาว คำนวณโดย power iteration จนกว่าจะ converge (1000 รอบ)

π = π · P    ← stationary dist satisfies this equation

ตัวอย่างจาก matrix ข้างต้น:
π ≈ [0.622, 0.323, 0.055] → Bull 62%, Bear 32%, Crisis 5%

muScale — proportional
drift = μ̂ × muScale

EM Equity, DM Equity, Thematic
muOverride — absolute
drift = muOverride (ค่าสัมบูรณ์ ต่อเดือน)

กลไก — Delta Rebalancing

Rebalancing เกิดขึ้น หลังจาก fee deduction และ premium contribution ในแต่ละเดือนที่กำหนด (monthly, quarterly, annual)

การ rebalance ไม่ได้ขายทุกกองทุนแล้วซื้อใหม่ทั้งหมด แต่ใช้วิธี Delta Rebalancing — ซื้อหรือขายเฉพาะส่วนที่ต้องปรับในแต่ละกอง:

1. ประเมินมูลค่าแต่ละกองที่ BID price → คำนวณ total
2. คำนวณ delta = targetValue − currentValue ต่อกอง
   • delta < 0 → กอง overweight: ขายส่วนเกินที่ BID price
   • delta > 0 → กอง underweight: ซื้อส่วนขาดที่ Offer price
   • |delta| ≈ 0 → กองที่ตรง target: ไม่มี transaction

ผลสุทธิ: สัดส่วนกลับมาตรง target โดยจ่าย spread เฉพาะ delta เท่านั้น
ตัวอย่าง Concrete (3 กองทุน)

Portfolio หลัง market move / target 33%/33%/34%: 

// Fund A: 100 units × bid 1.19 = 119.00  (overweight)
// Fund B: 100 units × bid 0.99 =  99.00  (underweight)
// Fund C: 100 units × bid 1.09 = 109.00  (underweight)
// total = 327.00

// targetA = 327×0.33 = 107.91  → delta = −11.09 → sell 9.32 units ที่ bid 1.19
// targetB = 327×0.33 = 107.91  → delta = +8.91  → buy  8.82 units ที่ offer 1.01
// targetC = 327×0.34 = 111.18  → delta = +2.18  → buy  1.96 units ที่ offer 1.11

// Fund A ขายแค่ 9.32 units (จาก 100) — ไม่ได้ขายทั้งหมด

หลังจากรัน N scenarios แล้ว จะคำนวณ percentile ที่แต่ละเดือน โดย sort ค่า portfolio value ของทุก scenario ณ เดือนนั้น แล้วเลือก index ที่ตรงกับ percentile ที่ต้องการ

P_p[m] = sorted_values[floor(p/100 × (N−1))]

const meanSeries = new Array(months).fill(0);
for (const s of allSeries) {
  for (let m = 0; m < months; m++) meanSeries[m] += s[m];
}
for (let m = 0; m < months; m++) meanSeries[m] /= N;
IRR — Internal Rate of Return

ใช้ IRR แทน CAGR เพราะ Unit Linked จ่ายเบี้ยเป็นงวด (DCA) CAGR สมมติว่าลงทุนทั้งหมดตั้งแต่วันแรก ซึ่งทำให้ตัวเลขต่ำกว่าจริงอย่างมีนัยสำคัญ

หาค่า r_monthly ที่ทำให้:
Σ_t premium · (1 + r)^(T − t) = finalValue

IRR_annual = (1 + r_monthly)^12 − 1

วิธีหา: Binary search 100 iterations (converges ใน <1μs)
✓
IRR รองรับทุก payment mode: monthly, quarterly, semi-annual, annual และถูกต้องสำหรับ DCA ที่จ่ายในช่วงเวลาต่างๆ

Seeded PRNG — Mulberry32

Simulation ใช้ Mulberry32 — PRNG แบบ 32-bit ที่ให้ sequence แบบ deterministic จาก integer seed ใดๆ แทนการใช้ Math.random() ของ browser ซึ่งไม่มี seed และ reproduce ไม่ได้ 

// simulation.js — mulberry32 PRNG
function mulberry32(seed) {
  return function() {
    seed = (seed + 0x6D2B79F5) | 0;
    let t = Math.imul(seed ^ (seed >>> 15), 1 | seed);
    // ... → ค่า float uniform [0, 1)
  };
}

ข้อดีเหนือ Math.random(): ให้ output เดิม 100% เมื่อใช้ seed เดิม, เร็ว (~3 ns/call), pass BigCrush statistical test suite

Per-Scenario Independence — seed + index

แต่ละ scenario สร้าง PRNG instance ของตัวเองโดยใช้ seed + scenarioIndex ทำให้ scenario แต่ละตัวมี sequence อิสระจากกัน และ reproducible — เพิ่ม N scenarios ทีหลังก็ไม่กระทบ sequence ของ scenarios ที่มีอยู่แล้ว 

// app.js / runMonteCarlo() — scenario loop
for (let i = 0; i < N; i++) {
  const rng = mulberry32(seed + i); // seed+0, seed+1, seed+2, …
  const series = runScenario({ ..., rng });
  // scenario i จะ identical ทุกครั้งที่ใช้ seed เดิม
}
Property:  f(seed, i) → sequence ที่ deterministic
seed เดิม + index เดิม = ผล Monte Carlo เดิมทุก bit
Shared Seed ข้าม Rebalancing Modes

เมื่อ tool รัน 4 rebalancing frequencies พร้อมกัน (none / monthly / quarterly / annual) ทั้ง 4 mode ใช้ seed เดียวกัน ทำให้ random shocks ของแต่ละ scenario เหมือนกันทุกประการ — ความแตกต่างของผลลัพธ์สะท้อนเฉพาะผลของ rebalancing ไม่ใช่ความผันแปรจากการสุ่ม 

// app.js — seed เดียวส่งให้ทุก mode
const seed = Date.now(); // สร้างครั้งเดียวก่อน loop
for (const mode of ['none', 'monthly', 'quarterly', 'annual']) {
  allResults[mode] = await runMonteCarlo({ ..., rebalanceMode: mode, seed });
}
✓
วิธีนี้ทำให้ตาราง "เปรียบเทียบความถี่ rebalancing" เป็น controlled experiment — variance จากการสุ่มถูก eliminate ความต่างที่เห็นในตารางคือ causal effect ของ rebalancing ล้วนๆ

Limitations
⚠️
Stationary Parameters
μ และ σ ถูก estimate จากข้อมูลอดีตและคงที่ตลอด simulation ในความเป็นจริง return distribution เปลี่ยนไปตามรอบเศรษฐกิจ
⚠️
Log-Normal Distribution
โมเดลสมมติว่า log-return เป็น Normal distribution แต่ return จริงมักมี fat tail (extreme events บ่อยกว่า Normal) ทำให้ downside scenarios อาจ underestimate
⚠️
Constant Correlation
Correlation matrix ระหว่างกองทุนถูก estimate ครั้งเดียวและคงที่ ในความเป็นจริง correlation มักเพิ่มขึ้นในช่วง market crash
⚠️
ยังไม่มีการหัก Fees
v1 ยังไม่ implement fee deductions (admin fee, COI, premium charge) ผลลัพธ์จึงสูงกว่าความเป็นจริงที่มี fee ทุกประเภท Fee hook (applyFees()) เตรียมไว้สำหรับ v2
⚠️
ข้อมูลประวัติศาสตร์เป็นเพียงการประมาณ
ผลการดำเนินงานในอดีตไม่ได้รับประกันผลในอนาคต Simulation นี้เป็น เครื่องมือช่วยวางแผน ไม่ใช่การพยากรณ์
